## 2. 需求项详述

### A1 · 接通法则选择入口
- **用户故事**：As a 理财新手，I want 在法则页点选一个法则后方案页立即展示该法则的分桶，so that 我能立刻看到策略落地效果，而不是永远看到"请先选择理财法则"。
- **优先级**：P0（阻塞全部下游）
- **验收标准**
  - Given 用户在 RulesPage 点击某条法则卡片，When 点击事件触发，Then `useAppStore.setSelectedRule(id)` 被调用且 `selectedRuleId` 更新。
  - Given `selectedRuleId` 已更新，When 路由切到 `/plan`，Then PlanPage 读取该 rule 并 `applyRule` 生成分桶，不再显示"请先选择理财法则"空态。
  - Given 用户后续切换另一条法则，When 再次点击，Then PlanPage 分桶随新法则实时重渲染（依赖 `PlanPage.tsx:27-30` 的 `useMemo` 已按 `rule` 重算，无需改）。
- **影响文件**：`pages/RulesPage.tsx`、`stores/useAppStore.ts`（已存在，仅调用）、`pages/PlanPage.tsx`（已基本就绪）。
- **衔接方式**：在 `RulesPage.tsx` 的 `RULES.map` 卡片容器加 `onClick={() => setSelectedRule(r.id)}`，并引入 `const setSelectedRule = useAppStore((s) => s.setSelectedRule)`；可顺带用 `selectedRuleId === r.id` 高亮当前选中卡片。PresetPicker 维持只写 `profile.preset` 不变（它影响推荐排序，与"选中法则"是两件事）。

### A2 · 修 A/B 对比末值（按法则实际投资桶月额复利）
- **用户故事**：As a 对比型用户，I want 切换对比的两条法则时复利末值随之变化，so that A/B 对比真实反映不同法则的增值差异。
- **优先级**：P0
- **验收标准**
  - Given 两条收入级法则（如 `4321` 与 `50-30-20`），When 在 `ComparePanel` 切换，Then 二者 `futureValue` 不再相等，且差异由各自投资桶月额决定。
  - Given 某投资级法则，When 参与对比，Then `futureValue` 基于"该法则投资桶月额"（= 投资池口径，见 C9）复利，而非 `monthlyDisposable`。
  - Given `buildPlanView` 被调用，When 返回 `PlanView`，Then `futureValue` 恒等于 `compoundSeries(该法则投资桶月额合计, 年化, investHorizonMonths)` 末点。
- **影响文件**：`features/plan/metrics.ts:30-40`（改 `futureValue` 计算）、`budget/engine.ts`（`compoundSeries` 复用）、需新增"投资桶识别"辅助函数。
- **衔接方式**：在 `metrics.ts` 内新增 `investmentMonthlyOf(rule, profile)`：
  - 对 `applyRule(profile, rule)` 生成的 `Bucket[]`，按"增值/储蓄类"判定汇总月额。
  - 增值类 `bucketKey` 集合（建议，待设计确认）：`invest, save, saving, grow, equity, stock, index, core, satellite, bond, ltbond, mtbond, commodity, tbill, principal, sprint`；消费类（不计入）：`living, need, want, play, edu, give, culture, food, cash, protect, safe, insurance, surprise`。
  - 单法则：`futureValue = compoundSeries(investmentMonthlyOf(rule, profile), annualRateFor(risk), months)`。
  - 注意：VisualizerPage 自身（`VisualizerPage.tsx:8` 的 `RATE` 表）与 `metrics.ts:20` 的 `annualRateFor` 年化口径不一致，本项不强制统一，但**建议在 PRD 评审中立项统一**（见 §4 风险 R5）。

### A3 · 复利可视加空态守卫
- **用户故事**：As a 用户，I want 档案未完善或收支为零时看到"请先完善档案"提示，so that 不会对着一条 0 值平线误以为已生成规划。
- **优先级**：P0
- **验收标准**
  - Given `profile` 为 null，When 进入 VisualizerPage，Then 显示"请先完善财务档案"空态（现状已满足）。
  - Given `profile` 存在但 `monthlyDisposable(profile) <= 0`（收入为 0 或支出≥收入），When 进入 VisualizerPage，Then 同样显示"请先完善档案 / 收支数据"空态，**不渲染 0 值曲线**。
  - Given 档案完整且 `monthlyDisposable > 0`，When 进入页面，Then 正常渲染复利曲线。
- **影响文件**：`pages/VisualizerPage.tsx:15-34`（在 `model` 生成后补充 `monthly <= 0` 守卫）。
- **衔接方式**：复用 `monthlyDisposable` 已有导入；守卫判定应放在 `useMemo` 之后，将"无可用月投资额为 0"与"无档案"归并为同一空态分支（文案可微调为"请先完善收入与支出"）。

### B4 · 顶栏 macOS 风格
- **用户故事**：As a 桌面用户，I want 应用拥有无边框 + 红黄绿窗口按钮的 macOS 风格顶栏，so that 视觉更现代、与产品调性一致。
- **优先级**：P1
- **验收标准**
  - Given `electron/main.ts` 启动窗口，When `BrowserWindow` 创建，Then 配置 `frame: false`（完全无边框），且保留 `titleBarStyle` 不设置（自定义）。
  - Given 渲染进程新增 `TitleBar.tsx`，When 页面加载，Then 顶部显示左侧三个圆形窗口按钮（红/黄/绿）+ 中间可拖拽区 + "FirstBucket" 文字。
  - Given 用户点击红/黄/绿，When 点击触发，Then 分别经 `ipcRenderer.invoke('window:close|minimize|toggleMaximize')` 通知主进程执行对应操作。
  - Given 主进程注册 `ipcMain.handle('window:minimize'|'window:close'|'window:toggleMaximize')`，When 收到，Then 调用 `win.minimize() / win.close() / win.isMaximized()?win.unmaximize():win.maximize()`。
  - Given 所有页面（`page-wrapper` / `main-content`），When 原生标题栏移除后，Then 顶部留出 ≥28px 的 padding 容纳 TitleBar，内容不被遮挡。
- **影响文件**：`electron/main.ts:53-65`、`electron/preload.ts`（新增 `window` 安全 API）、新增 `src/renderer/components/TitleBar.tsx`、各页面布局（或统一在 `App.tsx` 的 `app-shell` 层挂载 TitleBar + 调整 `styles.css` 的 `main-content` 顶部 padding）。
- **衔接方式**：
  - preload 经 `contextBridge` 暴露 `window.{minimize, close, toggleMaximize}`（保持与现有 `FirstBucket.profile/bucket/db` 同结构，禁止直接 `require`）。
  - TitleBar 拖拽区用 CSS `-webkit-app-region: drag`，按钮区用 `-webkit-app-region: no-drag` 以免吞掉点击。
  - TitleBar 建议在 `App.tsx` 的 `app-shell` 顶层挂载（与 `Nav` 并列），一次性解决所有页面顶部适配，而非逐页改。
  - 平台注意：当前运行平台为 win32；`frame:false` 在 Windows 下需额外保证窗口阴影/圆角与双击标题栏最大化（见 §4 风险 R1）。

### B5 · 徽章重构为数据达成型
- **用户故事**：As a 用户，I want 徽章基于真实财务指标解锁，so that 成就反映我的实际进展，而不是"点开功能就亮"。
- **优先级**：P1
- **验收标准**
  - Given `computeBadges` 输出，When 计算，Then 返回 5 个徽章：`surplus`(月结余为正) / `emergency`(备用金达标) / `goal`(已设储蓄目标) / `plan`(生成过方案) / `classify`(完成收支分类)。
  - Given `plan` 徽章，When 判定，Then 以"已保存方案"为准（`buckets.length > 0`，来自 `useAppStore.buckets`），而非 `!!selectedRuleId`。
  - Given `classify` 徽章，When 判定，Then 以"完成收支分类"为准（见开放问题 O3 的判定口径），移除原 `onboard`（完成引导即亮）徽章。
  - Given 某指标未达成，When 渲染，Then 该徽章呈未点亮（灰色）态，不误导。
- **影响文件**：`features/profile/health.ts:29-43`（重写 `computeBadges`）、`pages/ProfilePage.tsx:50`（改 `hasPlan` 来源）、`features/review/ReviewList.tsx:25` 与 `features/report/planReport.ts:33`（同步 `hasPlan` 口径）。
- **衔接方式**：保留 `Badge` 接口；`surplus`/`emergency`/`goal` 逻辑基本沿用现有 `computeBadges`（已属数据驱动）；主要改动为 `plan` 来源切换 + `onboard`→`classify` 替换。`classify` 的达成口径需 O3 拍板后落地。

### C6 · 五步财务画像（替代现有 Onboarding）
- **用户故事**：As a 新用户，I want 用五步向导录入收入/支出/分类/画像/三桶，so that 完成后我直接拥有一个可用的财务画像并进入 Dashboard。
- **优先级**：P1
- **验收标准**
  - Given 用户首次进入（`onboarded=false`），When 引导启动，Then 展示 5 步：①收入 ②固定支出 ③弹性支出+智能分类 ④财务画像 ⑤三桶分配。
  - Given Step1，When 录入月薪税前 / 社保公积金扣除 / 其他收入，Then 自动算"到手月薪"= 税前 − 社保公积金 − 其他扣除 + 其他收入（写入 `monthlyIncome` 等字段；具体字段映射见 §4 R6）。
  - Given Step2，When 录入房租/房贷、伙食、交通通勤、水电物业网费、保险、子女教育赡养，Then 实时显示小计并写入 `fixedExpenses`。
  - Given Step4，When 计算，Then 展示"到手 − 固定 − 弹性 = 可支配余额"，数值与 `monthlyDisposable` 一致。
  - Given Step5 完成，When 点"完成"，Then `setOnboarded(true)` 并跳转到 `/`（Dashboard），`profile` 已落盘。
- **影响文件**：`components/Onboarding.tsx`（重写为五步，或新建 `features/onboarding/FiveStepWizard.tsx` 由 `App.tsx:120` 调用）、`@core/domain/user.ts`（`monthlyIncome` 等字段已存在，仅需确认税前/扣除字段，见 R6）、`stores/useAppStore.ts`（`updateProfile` 复用）。
- **衔接方式**：完全替代现有轻量 `Onboarding`，保留 `App.tsx` 中 `{!onboarded && <Onboarding …/>}` 触发机制；Step4/Step5 复用 `monthlyDisposable` 与 C8 的三桶分配函数。

### C7 · 智能支出分类系统
- **用户故事**：As a 用户，I want 输入"生病 800""演唱会 500"后系统自动归类，so that 我不必手动理解每个类别，且可拖拽确认、标注频率、下次自动记忆。
- **优先级**：P1
- **验收标准**
  - Given 用户输入弹性开支项文本+金额，When 提交，Then 系统按关键词匹配归类（生病/药/医院→医疗应急；演唱会/电影/旅游→娱乐消费；人情/聚餐/份子→社交维系；手机/维修/家电→家居维修）。
  - Given 归类结果，When 展示，Then 用户可拖拽调整类别，且调整后即时写回该笔分类。
  - Given 每笔支出可标"一次性/月度/年度"，When 标注为一次性大额，Then 月均 = 金额 ÷ 12 摊入月度弹性支出。
  - Given 用户将某关键词调整为某类别，When 下次输入含该关键词的支出，Then 自动采用上次归类（偏好记忆，存本机）。
- **影响文件**：新增 `features/classify/` 模块（关键词词典 + 匹配器 + 偏好记忆 store）、`components/Onboarding.tsx`/五步向导 Step3 嵌入、可能新增 `variableExpenses` 明细结构（见 R6）。
- **衔接方式**：分类结果最终汇入 `profile.variableExpenses`（弹性支出小计）与 C8 三桶；偏好记忆存 `localStorage`（与现有 `fb_*` 模式一致）或并入 profile JSON。关键词词典初版由产品提供，后续可扩展。

### C8 · 三桶分配
- **用户故事**：As a 用户，I want 系统基于可支配余额自动分三桶，so that 我清楚哪些是保命钱、哪些是灵活钱、哪些可自由支配。
- **优先级**：P1
- **验收标准**
  - Given 可支配余额 `D = 到手 − 固定 − 弹性`，When 计算三桶，Then：
    - 备用金桶 = 覆盖 `3 × (固定 + 弹性) 月支出`，固定不动，仅限失业/重病/大额意外支取；
    - 灵活应急金桶 = 在固定+弹性花完后可支取，不用于娱乐；
    - 自由支配桶 = 剩余全部（储蓄/投资/消费自由）。
  - Given 备用金桶金额，When 与 `D` 比较，Then 若 `D < 备用金目标`，则备用金桶 = `D` 中尽可能多的部分，并在 UI 标注"备用金未达标"。
  - Given 三桶结果，When 展示，Then 三桶金额合计 = `D`（不溢出、不短缺）。
- **影响文件**：新增 `features/budget/threeBuckets.ts`（纯函数 `allocateThreeBuckets(profile)`），五步向导 Step5 与 Dashboard 复用。
- **衔接方式**：三桶是"可支配余额"的再分配，与 C6 Step4 的 `monthlyDisposable` 同源；备用金目标复用 `health.ts` 已有的 `3 × fixedExpenses` 思路，扩展为 `3 × (fixed + variable)`。三桶本身不写入 SQLite 桶表（那是法则分桶），仅作为画像洞察展示。

### C9 · 投资级法则改基于投资池分配 ⚠️（按设计师建议，用户未单独拍板）
- **用户故事**：As a 用户，I want 投资级法则（如 100−年龄）只切"投资池"而不是整月收入，so that 投资配置比例落在该投的钱上，而非被生活支出稀释。
- **优先级**：P1（与 D10 绑定交付）
- **验收标准**
  - Given `rule.ts` 每个法则新增 `scope: 'income' | 'invest' | 'withdraw'`，When 定义，Then 6 个 `income` / 5 个 `invest` / 1 个 `withdraw`（见 §0 归类）。
  - Given 某投资级法则，When `applyRule` 执行，Then 基数不再是 `totalMonthlyIncome(profile)`，而是"投资池"金额（见 O1 口径）。
  - Given `four-percent`（withdraw），When 进入方案生成/A/B 对比，Then 被排除（不参与 `PlanPage` 与 `ComparePanel` 的法则下拉）。
- **影响文件**：`domain/rule.ts`（加 `scope` 字段 + 更新 `Rule` 接口）、`budget/engine.ts:7-17`（`applyRule` 改为 scope 感知）、`pages/RulesPage.tsx`（下拉/卡片可按 scope 分组或过滤 withdraw）、`features/compare/ComparePanel.tsx`（排除 withdraw）。
- **衔接方式**：`applyRule` 内部：`if (rule.scope === 'invest') base = investmentPool(profile); else base = totalMonthlyIncome(profile)`。`investmentPool` 的精确定义见 §3 / O1，需设计师最终拍板。**此项为 PRD 内标注的最高风险开放项**。

### D10 · 给 12 法则标 scope
- **用户故事**：As a 研发，I want 每条法则有明确 scope，so that 引擎能区分"切收入"还是"切投资池"并支撑组合。
- **优先级**：P1（与 C9 同批）
- **验收标准**
  - Given `RULES` 数组，When 逐项补 `scope`，Then 校验：6×income + 5×invest + 1×withdraw = 12，且 `getRule`/`recommend` 不受破坏。
  - Given 类型系统，When 编译，Then `Rule.scope` 为必填字面量联合类型，遗漏任一条编译报错。
- **影响文件**：`domain/rule.ts`（纯数据 + 类型补充），无逻辑改动。
- **衔接方式**：本项是 C9 的数据基础，二者合并 PR/提交。

### D11 · 多选组合（2~3 个法则混合方案）
- **用户故事**：As a 进阶用户，I want 选 2~3 个法则生成混合方案，so that 我能融合多个策略（收入层加权 + 投资层拆分），末值随组合变化。
- **优先级**：P2
- **验收标准**
  - Given 用户选中 2~3 个法则（可含多 income / 多 invest 混合），When 调用新增 `applyRules(ruleIds, profile)`，Then 返回合并后的 `Bucket[]` 与 `PlanView`，且 `futureValue` 随组合变化（不复用单一 `monthlyDisposable`）。
  - Given 多个收入级法则，When 在"收入层"合并，Then 按规范映射至统一类目后加权取均值（见 §3 数学说明），生活/必需/储蓄/保险等类目比例合理。
  - Given 多个投资级法则，When 在"投资层"合并，Then 对投资池做资产类目加权拆分（权益/固收/商品等）。
  - Given 组合含 `withdraw` 法则，When 选择，Then 拒绝或忽略（提领法则不进入组合）。
- **影响文件**：新增 `features/rules/combine.ts`（`applyRules` + 类目映射表 + 加权函数），复用现有 `applyRule`/`compoundSeries`；`pages/PlanPage.tsx` 与 `features/compare/ComparePanel.tsx` 增加多选 UI 与组合视图。
- **衔接方式**：`applyRules` **不重写** `applyRule`，而是多次调用后做"两层合并"：先算收入层各法则的归一化类目占比 → 加权 → 得各收入类目月额；投资池 = 收入层中"储蓄/投资"类目月额合计 → 再对各投资级法则的资产类目占比加权 → 得资产桶月额。末值 = 投资层全部桶月额之和做复利。详见 §3。

## 3. 算法说明（第 9 / 11 项）：为何分层组合不可直接平均百分比

### 3.1 问题本质：两套不同的"基数（分母）"
- **收入级法则**的 `allocations.pct` 是基于**整月收入**（基数 = `totalMonthlyIncome`）的百分比。例如 `4321`：生活 40% / 储蓄 30% / 投资 20% / 保险 10%（合计 100%，分母是收入）。
- **投资级法则**的 `allocations.pct` 是基于**投资池**（基数 = 投资池金额，通常是收入的 20%~40%）的百分比。例如 `100-age`：权益 60% / 固收 40%（分母是投资池，不是收入）。
- **特殊提领** `four-percent`：基于本金的提取率，不参与。

### 3.2 直接平均百分比为何错误（数学原因）
若把 `4321` 的"投资 20%（占收入）"与 `100-age` 的"权益 60%（占投资池）"简单平均成 40%，犯了两重错误：
1. **分母不同**：前者 20% 的分母是"收入"，后者 60% 的分母是"投资池"。把不同分母的百分比相加再除，量纲不成立（如同把"身高占门高 90%"与"体重占行李限重 30%"平均）。
2. **类目字典不统一**：`4321` 用 `living/saving/invest/insurance`，`50-30-20` 用 `need/want/save`。即便都是收入级，直接对 `pct` 求平均也会把"生活 vs 必需""储蓄 vs 储蓄投资"错配，产生语义错乱的桶。

### 3.3 正确做法：两层顺序流水线（非平均）
```
层1 收入层：对每个选中的 income 法则 → applyRule(收入基数) → 归一化到统一类目字典
            → 按权重加权平均 → 得到 [生活, 必需, 储蓄, 投资, 保险, …] 各类目月额
            → 投资池 = 储蓄类目月额 + 投资类目月额
层2 投资层：对每个选中的 invest 法则 → 以"投资池"为基数 → applyRule(投资池)
            → 归一化到统一资产字典(权益/固收/商品/现金) → 加权平均 → 资产桶月额
末值 = compoundSeries(层2全部资产桶月额之和, 年化, 期限)
```
- **统一类目字典**（收入层）：`生活/必需`、`想要/弹性`、`储蓄`、`投资/增值`、`保障/保险`、`教育/玩乐/其他`。
- **统一资产字典**（投资层）：`权益类`、`固收类`、`商品`、`现金/货基`。
- **权重**：组合内同层法则等权（v1.2 默认），后续可扩展为按 `applicability` 打分加权（待定，见 O5）。

### 3.4 单法则视图（C9）下的投资池口径
当只选一个 `invest` 法则（如 PlanPage 单视图），没有收入层可借力，投资池必须单独定义 → 见 **O1**（需设计师拍板的开放项）。建议默认：`投资池 = monthlyDisposable(profile)`（即假设用户将全部可支配余额投入），以保证单视图与组合视图在数学上连续；但该假设影响末值绝对值，务必确认。

## 4. 风险与开放问题

### 风险
- **R1 · `frame:false` 布局适配范围**：移除原生标题栏后，所有页面顶部需补 padding；`Nav` 侧边栏与 TitleBar 的视觉衔接、窗口圆角/阴影（尤其 Windows）、双击标题栏最大化、拖拽区误吞按钮点击，均需在 B4 内回归验证。建议 TitleBar 在 `app-shell` 顶层统一挂载，避免逐页改。
- **R2 · 投资级改口径对旧数据兼容性**：现有已保存 `buckets`（SQLite）是按"整月收入"算的；C9 上线后 `applyRule` 基数变化，旧桶会失真。因分桶为按需重算（`PlanPage` 实时 `applyRule`），旧 `buckets` 仅作"已保存方案"展示，建议：C9 上线后提示用户"重新生成方案"，或直接以新版 `applyRule` 覆盖。无需改 DB schema。
- **R3 · `four-percent` 提领法则的 UI 排除**：`RulesPage` 卡片、`ComparePanel` 下拉、`applyRules` 入参均需显式过滤 `scope==='withdraw'`，否则会出现"基于本金 4% 的投资桶"。
- **R4 · 组合引擎的末值口径漂移**：`metrics.ts` 与 `VisualizerPage.tsx` 年化表（`annualRateFor` vs `RATE`）不一致；组合视图若另起一套，可能出现"方案页末值 ≠ 可视页末值"。建议在 D11 内统一年化口径或显式文档化差异。
- **R5 · 三桶与法则分桶的概念重叠**：C8 三桶是"可支配余额再分配"（画像洞察），法则分桶是"收入/投资池分配"（方案）。需在产品文案上区分，避免用户混淆"我到底有几个桶"。
- **R6 · UserProfile 字段粒度不足**：现有 `user.ts` 只有 `monthlyIncome`（到手口径），C6 Step1 需要"税前/社保公积金扣除/其他收入"明细，C7 需要"弹性支出明细（含类别/频率）"。需在 `user.ts` 扩展字段（如 `grossIncome`、`socialInsuranceDeduction`、`expenseItems[]`），属向后兼容的 JSON 增量（与现有 v1.1 扩展模式一致）。

## 5. 第 9 / 11 项算法小结（给研发的速查）
- 投资级法则加 `scope:'invest'`，`applyRule` 基数改 `investmentPool(profile)`（O1 定）。
- 组合引擎 `applyRules` 走"收入层加权 → 得投资池 → 投资层加权拆分"两层流水线，**绝不对跨层百分比求平均**。
- 末值统一 = 投资层全部资产桶月额之和做 `compoundSeries`（复用 `engine.ts`）。
