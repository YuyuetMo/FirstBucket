## 2. 需求项详述

> 用户故事格式：As a … / I want … / so that …  
> 验收标准格式：Given … / When … / Then …  
> 优先级约定：P0=阻塞/用户痛点；P1=核心体验；P2=增强。

### R1 · 法则按人生阶段动态筛选 + 推荐随阶段变

- **用户故事**：As a 用户，I want 切到"已婚双职工""临退休"等人生阶段后法则库自动重排——推荐的置顶、适合的居中、不搭的灰显，so that 我不用在 12 条里自己挑，降低选择焦虑。
- **优先级**：P1
- **验收标准**
  - Given 每条法则新增 `lifeStages: string[]`（适配阶段标注），When `PresetPicker` 选中某阶段 `preset`，Then `RulesPage` 对所有法则按"推荐 > 适合 > 低关联"三档重排（非静态初始序）。
  - Given 当前阶段 `preset`，When 某法则满足 `preset ∈ rule.lifeStages` 且 `rule.id ∈ getPreset(preset).recommendRules`，Then 该法则归入"推荐"档并置顶，显示「为你推荐」标签（标签随 `preset` 重新计算，非写死）。
  - Given 某法则满足 `preset ∈ rule.lifeStages` 但不在 `recommendRules`，When 渲染，Then 归入"适合"档、正常展示。
  - Given 某法则 `preset ∉ rule.lifeStages`，When 渲染，Then 归入"低关联"档，按补充 A 灰显虚线、仍可见可点。
- **影响文件**：`ROOT/src/renderer/@core/domain/rule.ts`（加 `lifeStages` 字段 + 12 法则逐项标注）、`ROOT/src/renderer/pages/RulesPage.tsx`（按阶段三档重排渲染）、`ROOT/src/renderer/features/preset/presets.ts`（复用既有 `recommendRules`/`weights`）、`ROOT/src/renderer/styles.css`（低关联灰显虚线、档位分隔）。
- **衔接方式**：`lifeStages` 为 `Rule` 新增**必填**字面量数组，元素取值 = `PresetId`（`student|newgrad|single|dualincome|family|preretire`）。`RulesPage` 内部按上述派生逻辑分组排序后渲染；`getPreset(preset)?.recommendRules` 已存在且随 `preset` 变化，天然满足"非静态"。**5 条核心原则**（`RulesPage.tsx:37-84` 的 `principles` 硬编码数组）的处理见 §4 开放问题 O1。

**R1 拟议 `lifeStages` 全量标注（产品经理初稿，待设计师最终拍板，见 O2）：**

### R5 · 复利图表旁自动说明文案（数据来源+算法含义）【补充 C 归并】

- **用户故事**：As a 用户，I want 复利图上方有一行浅色小字说明"这笔数怎么来的"，so that 我理解曲线为何有这么多点、基于哪条法则的哪个月额、按什么年化。
- **优先级**：P2
- **验收标准**
  - Given `PlanPage`（方案 Tab）或 `VisualizerPage` 渲染复利图，When 图表上方，Then 显示固定一行浅色说明，文案形如：`基于「X」法则的投资桶月额 ¥xxx，按 Y% 年化复利计算。每一点代表当年年末资产总额。切换法则时曲线自动重算。`

### 3.2 R8 详细设计（表单 / 校验 / 持久化 / 入库参与组合）

**3.2.1 数据模型（与 `Rule` 兼容）**
```ts
// rule.ts 内：Rule 加可选标记，Allocation 完全复用
export interface Rule {
  id: RuleId | string;          // 自定义用 'custom-<timestamp>'
  name: string;
  description: string;
  tags: string[];
  allocations: Allocation[];     // 同结构：{ label, pct, bucketKey, color }
  scope: 'income' | 'invest';   // 自定义仅可选 income/invest（不含 withdraw）
  applicability: (p: UserProfile) => number; // 自定义给常量函数 () => 60
  custom?: boolean;              // 标记自定义
}
export interface CustomRuleInput {
  name: string;
  scope: 'income' | 'invest';
  allocations: { label: string; pct: number; bucketKey: string; color: string }[];
}
```

**3.2.2 持久化方案（沿用 `fb_*` localStorage 模式，不新增持久化层）**
- key：`fb_custom_rules`，value：`CustomRuleInput & { id; custom: true }` 的 JSON 数组。
- `customRules.ts`：
  - `loadCustomRules(): Rule[]` — 读 localStorage 解析；首次为空返回 `[]`。
  - `saveCustomRule(input: CustomRuleInput): Rule` — 生成 `id='custom-'+Date.now()`，补 `custom:true`、`applicability:()=>60`、`tags:['自定义']`，push 进数组并写回 localStorage，返回该 `Rule`。
  - `updateCustomRule(id, input)` / `deleteCustomRule(id)` — 增删改数组后写回。
- **运行时注册**：`rule.ts` 的 `getRule(id)` 改为：先 `RULES.find`，未命中则查 `loadCustomRules().find`。因 `loadCustomRules` 是纯读 localStorage，无副作用，**不破坏引擎纯函数语义**（仅在解析入口触发，符合 v1.2 硬约束②"对引擎修改仅加法"）。

**3.2.3 表单与实时校验（R8 验收核心）**
- 动态桶列表：初始 2 行；每行 `桶名 input` + `比例% number` + 删除按钮；底部「＋ 添加桶」。
- 实时合计：`sum = allocations.reduce((s,a)=>s+a.pct,0)`。
- 校验：`sum !== 100` → 「保存」`disabled` 且提示 `当前合计 ${sum}%，还差 ${100-sum}%`（sum<100）或 `超出 ${sum-100}%`（sum>100）。
- `scope` 选择：收入级/投资级单选（决定该法则进组合引擎的收入层还是投资层）。
- 颜色：每行给默认调色板循环，用户不改也可存。

**3.2.4 入库后参与组合的机制**
- `getRule` 扩展后，`applyRules(ruleIds, profile)`（`combine.ts:89`）经 `ruleIds.map(getRule)` 自然解析到自定义法则，无需改签名。
- 自定义 `scope==='income'` → 进组合层1 收入层；`scope==='invest'` → 进层2 投资层；`withdraw` 不允许（表单不提供），故不会触发 `combine.ts:96` 的 withdraw 过滤冲突。
- `ComparePanel` 下拉、`RulesPage` 卡片渲染均通过 `getRule`/`RULES`+自定义列表取数，自动包含自定义法则，并打「自定义」标签（徽章样式复用 `.badge-gold` 或新增 `.badge-custom`）。
- 编辑/删除：在卡片「自定义」标签旁加「编辑」「删除」入口；删除仅 `deleteCustomRule`，不影响内置 12 法则，也不影响已保存方案（`buckets` 已落盘为具体金额，与法则定义解耦）。

## 4. 风险与开放问题

### 风险（R）
- **R1 · 自定义法则与组合引擎 scope 处理**：自定义法则的 `scope` 决定它进收入层还是投资层。若用户把"本应投资级"的法则标成 `income`，组合末值会失真。缓解：表单 `scope` 选项加一句话说明"收入级=切整月收入；投资级=切投资池"，并在保存时按 scope 走对应层（已在 §3.2.4 说明，引擎无需特判）。
- **R2 · lifeStages 标注的主观性**：12 法则×6 阶段的适配标注含产品主观判断，不同设计师可能不同。缓解：本文 §2 R1 已给初稿表，列为 **O2 待设计师拍板**；标注为 `Rule` 必填字段，编译期强制不漏。
- **R3 · What-if 参数隔离后基准数据的来源**：基准"方案"Tab 必须始终读 `useAppStore.profile`（真实五步画像），What-if 仅经 `tempProfile` 注入"方案"Tab 的 `effective`。风险点：若某处误把 `tempProfile` 当基准，则基准会被污染。缓解：R4 明确"方案"Tab 默认 `tempProfile ?? profile`，且切走 What-if Tab 时 `setTempProfile(null)`（沿用 `WhatIfPanel.tsx:50-53` 的切换清空逻辑，扩展为 Tab 切换也清空）。
- **R4 · VisualizerPage 基数与 R5 说明口径不一致（v1.2 遗留）**：当前 Visualizer 用 `monthlyDisposable`、PlanPage 用 `investmentMonthlyOf(rule)`。R5 说明写"基于「X」法则的投资桶月额"，若 Visualizer 不改基数则文字与曲线矛盾。缓解：R5 验收已要求 Visualizer 基数改 `investmentMonthlyOf(rule)`；同时建议 R5 文案在 Visualizer 未选法则时回退为"基于每月可支配 ¥x"（避免 null 崩溃）。
- **R5 · 三桶与法则分桶再混淆**：R6 把"三桶（备用金/灵活/自由）"与"法则投资桶"并排展示，用户可能混淆"我到底有几个桶"。缓解：R6 清单区用小标题明确分段——"保命钱（三桶洞察）"vs"投资分配（所选法则）"，沿用 v1.2 §3.5 的文案区隔。
- **R6 · 零新增依赖下的右键菜单/模态**：R3 右键菜单、R8 创建面板均须用原生 DOM + 唯一 CSS 实现，不能引组件库。已在衔接方式中限定为原生 `onContextMenu`/`绝对定位 div`/原生 `<dialog>` 或自定义遮罩。

### 开放问题（需设计师 / 用户最终拍板）
- **O1 · 5 条核心原则是否纳入 lifeStages 重排体系？** `RulesPage.tsx:37-84` 的 `principles` 是硬编码数组（含"先支付自己""50/30/20""建立紧急备用金""SMART 目标法""避免高利负债"），其中"先支付自己"用户指定全阶段、"smart-goal"指定适配单身白领，**暗示原则也应参与重排**。但将其结构化需把 `principles` 提升为有 `lifeStages` 的数据（可能并入 `rule.ts` 或新建 `principles.ts`）。待拍板：①原则永驻顶部"通用原则"块、不重排；或 ②原则也加 `lifeStages`、与 12 法则一起三档重排。**本文默认采用 ②（与用户示例一致），但具体映射见 §2 R1 拟议表，待确认。**
- **O2 · lifeStages 全量标注终稿**：§2 R1 表为产品经理初稿，需设计师确认每法则×每阶段的适配判定（尤其 `student`/`preretire` 边界）。
- **O3 · What-if 场景杠杆的复合口径**：年薪涨幅/通胀率按几年复合？`investHorizonMonths/12` 还是独立"推演年限"输入？大额支出时间按月序号还是年序号？失业月份如何折算进 `monthlyDisposable` 序列？需在 `scenario.ts` 落地前拍板（建议：均按 `investHorizonMonths` 为总期，涨幅/通胀按 `(1+r)^(months/12)` 复合，大额支出取"第 N 月"、失业取"前 M 月 income=0"）。
- **O4 · "通胀率"杠杆 ≠ 被排除的"M4 通胀开关"**：R4 的 What-if 通胀率是**单场景临时杠杆**（不持久、不全局生效），E 批排除的"M4 通胀开关"是**全局持久化开关**。二者语义不同，请在评审中明示以免误判为范围冲突。
- **O5 · 自定义法则的 `applicability` 打分**：本文给常量 `()=>60`，使其在 R1 重排中归"适合"档而非"推荐"档。是否应允许用户自定义适配阶段（加 `lifeStages` 到自定义模型）？待拍板，默认不给（保持表单简单）。
